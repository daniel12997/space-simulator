# Berry-Healy 2004 GJ8 Coefficient Generator

The companion code to [[sources/berry-healy-2004-gauss-jackson]]: Liam Healy's
Common Lisp generator that produces the named coefficient tables of the
paper (Tables 3, 4, 5, 6 plus Eqs 27 / 32 / 44 / 48 / 68 / 70). Authoritative
for re-deriving Apsis's GJ8 implementation when the C++ transcription needs
verification against an unambiguous algebraic reference rather than against
the paper's printed values (which are subject to OCR / typesetting drift in
PDF reprints).

## Files

- `coefficients.lisp` — Healy's original ANSI Common Lisp source, 2005-05-24.
  Preserved verbatim as a corpus archive of the generator as Healy
  distributed it. **Do not edit** — this is the historical record.
- `coefficients.py` — Python port (exact-rational `fractions.Fraction`
  arithmetic, `functools.lru_cache` for memoization). Function-by-function
  faithful translation of the Lisp **except** for `half_acceleration_in_sum`,
  which carries a corrected diagonal-alignment logic — see "Upstream Lisp
  bug" below.
- `coefficients-output.txt` — captured stdout from `python3 coefficients.py`.
  This is the canonical reference for downstream C++ transcription. All
  values are exact rationals.
- `verify.py` — verification harness. Cross-checks the Python output
  function-by-function against printed paper values at every named anchor
  (Eqs 27, 32, 44, 48, 68 last row, 70 last row; Tables 3, 4, 5, 6) and
  asserts the algebraic invariants (cumulative-sum recurrence,
  predictor↔corrector inversion, autoconvolution definition of Cowell, Eq
  67 re-derivation of `difference_to_ordinate`). Exits non-zero on any
  mismatch.

## Upstream Lisp bug

Healy's `half-acceleration-in-sum` (lines 73–79 of `coefficients.lisp`) has
a 20-year-old latent diagonal-alignment bug. The Lisp's `(if (= k j) ...)`
adds `+1/2` where the Lisp column index `k` equals the Lisp row index `j`,
but the paper's "current point" diagonal is at paper-`k` == paper-`j`,
which corresponds to Python `(row r, col r-1)` for `r = 1..N+1` and to
"no cell" for `r = 0` (the predictor row).

**Effect on the Lisp's Table 5 output**: 18 of 90 cells disagree with the
paper at p. 347. Specifically:

- 9 cells where `+1/2` was added at the wrong column (one column right of
  the paper diagonal, in mid-corrector rows).
- 9 paper-diagonal cells where `+1/2` was correctly required but never
  added (because the Lisp's off-by-one would have written to a column
  index outside the row's range).
- 1 spurious `+1/2` in the predictor row at column 0 (paper expects raw
  ordinate; no diagonal modification on the predictor).

**The paper's printed Table 5 was NOT produced by Healy's Lisp.** Either it
was generated by a different code path or hand-shifted by Healy /
Berry / the typesetter at publication time. The printed values are correct
per the math; the Lisp generator is wrong.

**Discovery**: the bug was found during the Apsis Phase 1A Batch D''
preparation when a clear-eyes reviewer (2026-05-06) cross-checked the
generator's output against the paper's printed Table 5 anchor values.
Specifically, paper Table 5 `(j=-4, k=-4) = +19087/89600` and
`(j=4, k=4) = -19087/89600` — both are corrector-current-point cells where
the `+1/2` belongs and the Lisp had not added it. The Python port (a
faithful function-by-function translation) had inherited the bug.

**Resolution**: the Python port's `half_acceleration_in_sum` (lines 126–158
of `coefficients.py`) was corrected to honour the paper's diagonal
convention; the Lisp file is preserved unmodified. The corrected output
matches the paper at all checked Table 5 anchors. See `verify.py` for the
test harness; run it with `python3 verify.py` to confirm.

## Use in Apsis

The Phase 1A Batch D'' (Berry-Healy 2004 ordinate-form Gauss-Jackson 8)
implementation transcribes Tables 5 and 6 from `coefficients-output.txt`
into `constexpr` C++ tables in `src/integrate/gj8_coeffs.h`. Because the
output is exact rationals, the implementer reduces each `p/q` to a
double at compile time with no precision-loss path beyond the unavoidable
`p/q` → `double` rounding (typically ~52 bits of precision retained).
The compile-time `static_assert` table-checksum tripwire is computed
against this output as the source-of-truth, NOT against the paper PDF.
